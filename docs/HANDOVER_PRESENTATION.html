Architektur-Übersicht

Frontend Architektur

  • SPA: Single Page Application mit React Router
  • State: Zustand für global, TanStack Query für Server
  • API Client: Typed fetch-wrapper mit error handling
  • Offline: Service Worker + IndexedDB Queue
  • PWA: Installierbar, Offline-fähig

Backend Architektur

  • 3 Backend-Optionen: Node, Vercel Serverless, Alerts-Service
  • Node Backend: Always-on, SQLite/Postgres, Jobs
  • Vercel Functions: Serverless, cold-start optimiert
  • Alerts Service: Dedicated, Express + Postgres + SSE

Data Layer

  • ORM: Drizzle mit Type-safe queries
  • Migrations: SQL-basiert, versioniert
  • Caching: Multi-layer (Memory, Redis, IndexedDB)
  • Queue: BullMQ für async Jobs

API Design

  • Contracts: Shared Zod schemas (Single Source of Truth)
  • Envelope: { status: "ok", data } / { error: {...} }
  • Versioning: URL-basiert (/api/v1/...)
  • REST: Standard REST mit semantic routes

Deployment

Production Setup

  • Frontend: Vercel (Edge Network)
  • Backend: Railway (Node Backend)
  • Database: Railway Postgres
  • Redis: Railway Redis
  • Alerts: Separate Railway Service

CI/CD

  • GitHub Actions: Test & Lint
  • Vercel: Auto-deploy PRs & main
  • Railway: Auto-deploy main branch
  • E2E Tests: Playwright in CI

Monitoring

  • Performance: Web Vitals tracking
  • Errors: Error Boundary + Logging
  • Analytics: Optional (Privacy-first)
  • Uptime: Health-Check Endpoints

Security

  • API Keys: Env Variables, nie im Frontend
  • CORS: Configured für production domains
  • Rate Limiting: Per IP und Route
  • Input Validation: Zod auf allen Ebenen

📦 Monorepo: Workspace-Setup mit pnpm, shared contracts zwischen Frontend und Backend
